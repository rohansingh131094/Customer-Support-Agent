What this is

A focused prototype for Bookly, a fictional online bookstore, demonstrating how an AI support agent can handle order status, returns, and policy questions through identity checks, scoped SOPs, mocked tools, and real-time streaming.

The Thesis

Customers expect instant, accurate, empathetic support — every time. Human teams struggle to deliver that consistently at scale. The question isn't whether to automate. It's how.

Great CX is reasoning
within constraints —
not scripted
deflection.

An effective support agent should understand what the customer needs, gather the minimum required context, use trusted systems of record, and know when not to act.

How this build reflects it
Goal-based SOPs
Each workflow defines the goal, required context, allowed tools, and escalation rules for that customer intent. This gives the agent flexibility to handle varied customer phrasing without relying on a brittle step-by-step script.
Bounded autonomy
The agent can reason through customer requests, but only inside defined boundaries: identity before order access, eligibility before returns, and confirmation before irreversible actions.
Grounded responses
The agent reduces hallucination risk by grounding customer-specific answers in Bookly tools instead of relying on model memory. It asks for missing context, calls the relevant tool, and responds from the result.

Key Engineering Decisions

Every choice made deliberately — and why.

Direct Tool Loop, No Agent Framework
Choice Used Anthropic SDK directly with no abstraction layer
Tradeoff More code and responsibility for orchestration, errors, and retries
Rationale Exposes the tool loop, streaming, and error paths clearly — no magic hiding the behavior
Cheap Router, Strong Executor
Choice Haiku classifies intent, Sonnet handles reasoning and action
Tradeoff One extra model call per turn
Rationale Keeps prompts scoped and reduces cost and latency versus using Sonnet for everything
Goal-Based SOPs, Scoped by Intent
Choice Each intent gets its own goal-oriented SOP and allowed tool subset
Tradeoff More prompt and config surface to maintain as intents grow
Rationale Safer behavior, fewer irrelevant tools in scope, and better hallucination control

Scaling to Production

Five decisions, ordered by when they matter.

Ship Blockers
Persistent Sessions
Replace in-memory state with Redis or DynamoDB — survives restarts, enables horizontal scaling, and is a hard requirement before multi-instance deploy.
Structured Observability
Trace every intent decision, tool latency, and token spend — without this, production failures are invisible until a customer notices.
Cost Efficiency
Prompt Caching
Cache stable system prompts and SOPs to reduce repeated input-token cost on every turn at scale.
Quality
RAG Knowledge Base
Vector search over policy docs — improves accuracy on edge-case queries and lets policy content update without code changes.
LLM-as-a-Judge Evals
Automated eval pipeline scores responses on compliance, accuracy, and tone after every prompt change.
